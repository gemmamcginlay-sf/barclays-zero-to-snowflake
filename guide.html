Snowflake at Barclays

5 min — verbal
Snowflake is a compute engine within EDP. Your data lives in S3 as Iceberg. Snowflake reads it without moving it.
  • EDP architecture: S3 → Iceberg → Snowflake (compute) → Consumers
  • Multi-tenant: each BU gets isolated databases, warehouses, roles
  • Pay only when queries run — auto-suspend = zero idle cost
  • Approved patterns: dbt, Dynamic Tables, Tasks, Cortex AI, Streamlit
⚠️
Demo vs Production:

This demo uses Snowflake-native tables and plain-text synthetic data for clarity. In production: data sources from S3/Iceberg (no copy), sensitive fields arrive pre-tokenised (Design Principle #7), and schema names follow EDP tiers (Raw/Base/Prepared).

Snowflake is not a data lake and not a separate copy of data. It's compute that sits on top of what already exists in S3.

Pipelines

25 min — live SQL + dbt

sql/03_pipelines.sql

Six transformation patterns. Think of this as the menu — pick the right tool for each job.
ℹ️
EDP Principle #11:

dbt is the strategic standard. Dynamic Tables, Tasks, and Procedures are all supported.

4a. Views — logical layer

Create the STAGING.PAYMENTS_CLEANSED view — adds VALUE_BAND classification
Query it to show RETAIL vs CORPORATE vs LARGE VALUE breakdown
Views add business logic without storing a copy. Always fresh. Zero cost. This is your STAGING layer.

4b. Dynamic Tables — auto-refreshing

Create PREPARED.DAILY_SUMMARY with TARGET_LAG = '1 hour'
Query it immediately — show pre-aggregated daily KPIs
Declare WHAT you want, not WHEN to refresh. Snowflake handles the scheduling.
Materialised results that stay fresh automatically. Ideal for dashboards and operational summaries.

4c. dbt — version-controlled pipelines

Show the dbt project structure in Snowsight (or Cortex Code)
Walk through: source → stg_payments → mart_payment_kpis → mart_sla_compliance
Show schema.yml tests: not_null, unique, accepted_values
Query the built marts — show KPIs and SLA compliance (COMPLIANT / AT RISK / BREACH)
Same SQL we've been writing. dbt adds version control, testing, docs, and lineage.
dbt is the EDP standard for repeatable transformations. Code-reviewed, tested, traceable — not scripts on someone's laptop.

4d. Streams + Tasks — event-driven (live demo)

Step 1: Show baseline PENDING count
Step 2: Create Stream + Task
Step 3: INSERT 5 rows into PAYMENT_STATUS_UPDATES (simulate upstream)
Step 4: Query the Stream — show METADATA$ACTION (built-in CDC)
Step 5: EXECUTE TASK — fire it manually
Step 6: Show stream is empty, PENDING dropped by 5
No Kafka, no Debezium, no Airflow. Event-driven in 4 SQL statements. Zero cost when no data.
Streams capture changes automatically. Tasks fire only when data exists. The full event-driven pipeline without external tools.

4e. Notebooks — Python + SQL together

notebook/payments_operations_analysis.ipynb

Open Snowsight → Projects → Notebooks → Import the pre-built notebook from the repo (or create from Git)
Set compute to BARCLAYS_DEMO_NOTEBOOK_POOL (container runtime — full pip access)
Run Cell 1 (SQL): Payment volume by type and status — show tabular results inline
Run Cell 2 (Python): Converts SQL output to Pandas, computes failure rates — show the pivot table
Run Cell 3 (SQL + Python): Daily failure trend query feeds into an Altair line chart — show the red trend line
Run Cell 4 (Python): Bar chart comparing failure rates across SWIFT/BACS/CHAPS — highlight which types are worst
Run Cell 5 (SQL + Python): Processing time percentiles (P50 vs P95) as grouped bars — show latency differences
SQL and Python in the same document. Each cell runs on Snowflake compute — no data download, no local setup. The Python cells reference SQL cell results by name.
Show deployment: click Schedule → set to daily → assign warehouse — notebook becomes a scheduled job
Notebooks bridge SQL and Python in a single governed environment. Deploy them as scheduled jobs for pipelines that need data science logic alongside SQL transforms.

4f. Stored Procedures — control flow

Create and CALL FLAG_AGED_FAILURES() — show IF/ELSE logic returning a count
For logic that needs branching, loops, or error handling. Supports SQL, Python, Java, JavaScript.

Snowflake Intelligence CoWork

15 min — live UI + chat

sql/05_intelligence.sql (for the analytics view)

Everything we've built so far requires SQL. What about people who don't write SQL? That's what CoWork solves.

6a. Create the Semantic View (live)

AI & ML → Analyst → Semantic Views → Create with Autopilot
Select V_PAYMENTS_ANALYSIS as the source
Name it SV_PAYMENTS_BARCLAYS — let Autopilot generate the definitions
Review the generated metrics and dimensions — show how business language maps to columns
The Semantic View defines trusted business definitions. This is what ensures the AI generates correct queries, not hallucinated ones.
Semantic Views are the governance layer for AI. They define what "total volume" or "success rate" means so every answer is consistent.

6b. Create the Agent (live)

AI & ML → Agents → Create Agent
Name: BARCLAYS_PAYMENTS_AGENT
Add tool: Query Structured Data → select SV_PAYMENTS_BARCLAYS
Save — show how quick it is to go from data to conversational interface
That's it. A conversational analytics agent over your payments data. Built in 60 seconds.
Creating an Agent takes seconds. It combines the Semantic View with conversational AI — no code, no deployment.

6c. Live Agent Demo — Conversational Analytics

Open CoWork → select BARCLAYS_PAYMENTS_AGENT
Ask: "What is the total payment volume this month?"
Follow up: "Which payment type has the lowest success rate?"
Drill down: "Show me the SWIFT failures by region"
Trend: "Show the trend of CHAPS payments by week"
Reasoning: "Why might APAC have lower success rates?"
Same data we've been querying all session. Now accessible to anyone via a conversation.
Agents maintain context across a conversation. Follow-ups, drill-downs, reasoning — all governed by the Semantic View. This is Snowflake Intelligence.
