Querying Data

12 min — live SQL

What this covers: Hands-on SQL — from simple selects through aggregations, window functions, instant charting, and Time Travel.

Why it matters: If you already write SQL, you can be productive in Snowflake immediately. No new language. Time Travel eliminates the need for manual backups or restore tickets.

3a. Explore

sql
SELECT * FROM PAYMENTS LIMIT 10;
Browse and preview any table you have access to — no exports, no tickets.

3b. Aggregate + Chart

sql
SELECT
    PAYMENT_TYPE,
    COUNT(*) AS TXN_COUNT,
    SUM(AMOUNT) AS TOTAL_VALUE,
    COUNT(CASE WHEN STATUS = 'FAILED' THEN 1 END) AS FAILED,
    ROUND(AVG(PROCESSING_TIME_MS), 0) AS AVG_MS
FROM PAYMENTS
GROUP BY PAYMENT_TYPE
ORDER BY TOTAL_VALUE DESC;

Click "Chart" tab → instant bar chart

Standard GROUP BY — same SQL you'd write anywhere. Built-in charting removes the need to export to Excel for a quick visual.

3c. Window function

sql
SELECT
    PAYMENT_DATE::DATE AS DAY,
    SUM(AMOUNT) AS DAILY_VALUE,
    SUM(SUM(AMOUNT)) OVER (ORDER BY PAYMENT_DATE::DATE) AS RUNNING_TOTAL
FROM PAYMENTS
WHERE STATUS = 'COMPLETED'
GROUP BY DAY ORDER BY DAY;
Window functions work identically to Teradata/Oracle/SQL Server. Running totals, rankings, and moving averages are all single-query operations.

3d. Time Travel

sql
-- Make a change
UPDATE PAYMENTS SET STATUS = 'CANCELLED'
WHERE STATUS = 'PENDING' AND AMOUNT < 1000;

-- Query 30 seconds ago (talk while waiting)
SELECT STATUS, COUNT(*)
FROM PAYMENTS AT(OFFSET => -30)
GROUP BY STATUS ORDER BY COUNT(*) DESC;

-- Undo instantly
CREATE TABLE PAYMENTS_RESTORED CLONE PAYMENTS AT(OFFSET => -60);
ALTER TABLE PAYMENTS SWAP WITH PAYMENTS_RESTORED;
DROP TABLE PAYMENTS_RESTORED;
Snowflake retains historical versions automatically. Query any point in time (up to 90 days) and restore instantly — no support ticket needed.

Pipelines

25 min — live SQL + dbt

What this covers: Five transformation patterns — Views, Dynamic Tables, dbt, Streams+Tasks, and Stored Procedures. Each with a real example.

Why it matters: Most teams quickly need repeatable pipelines. This section gives you the full menu so you pick the right tool for each job, aligned with EDP standards.

ℹ️
EDP Principle #11:

dbt is the strategic transformation framework. Dynamic Tables, Tasks, and Stored Procedures are all supported.

4a. Views — always-current logical layer

sql
CREATE OR REPLACE VIEW STAGING.PAYMENTS_CLEANSED AS
SELECT *,
    PAYMENT_DATE::DATE AS PAYMENT_DAY,
    CASE WHEN AMOUNT < 5000 THEN 'RETAIL'
         WHEN AMOUNT < 100000 THEN 'CORPORATE'
         ELSE 'LARGE VALUE' END AS VALUE_BAND
FROM RAW.PAYMENTS;
Views add business logic without storing a copy. Consumers always see the latest data. Zero cost.

4b. Dynamic Tables — auto-refreshing materialisation

sql
CREATE OR REPLACE DYNAMIC TABLE PREPARED.DAILY_SUMMARY
    TARGET_LAG = '1 hour'
    WAREHOUSE = BARCLAYS_DEMO_WH
AS
SELECT PAYMENT_DAY, PAYMENT_TYPE, REGION,
    COUNT(*) AS TXN_COUNT,
    SUM(AMOUNT) AS TOTAL_VALUE,
    ROUND(100.0 * COUNT(CASE WHEN STATUS='FAILED' THEN 1 END) / COUNT(*), 2) AS FAIL_RATE_PCT
FROM STAGING.PAYMENTS_CLEANSED
GROUP BY PAYMENT_DAY, PAYMENT_TYPE, REGION;
Declare WHAT, not WHEN. Snowflake refreshes automatically when source changes. Ideal for dashboards and operational summaries.

4c. dbt — version-controlled SQL (EDP standard)

🔀
Live switch:

Open the dbt project in Snowsight or Cortex Code to walk through.

dbt wraps SQL in version control, testing, and documentation. The EDP-approved standard for repeatable transformations.

4d. Streams + Tasks — event-driven (live demo)

Full lifecycle: create the pipeline, simulate data arriving, watch the stream capture it, fire the task, prove the data updated.

sql — 6 steps
-- Step 1: Baseline
SELECT STATUS, COUNT(*) FROM RAW.PAYMENTS GROUP BY STATUS;

-- Step 2: Create Stream + Task
CREATE OR REPLACE STREAM RAW.STATUS_STREAM ON TABLE RAW.PAYMENT_STATUS_UPDATES;
CREATE OR REPLACE TASK RAW.APPLY_UPDATES
    WAREHOUSE = BARCLAYS_DEMO_WH  SCHEDULE = '5 MINUTE'
    WHEN SYSTEM$STREAM_HAS_DATA('RAW.STATUS_STREAM')
AS MERGE INTO RAW.PAYMENTS t USING RAW.STATUS_STREAM s
   ON t.PAYMENT_ID = s.PAYMENT_ID
   WHEN MATCHED THEN UPDATE SET t.STATUS = s.NEW_STATUS;

-- Step 3: Simulate data arriving
INSERT INTO RAW.PAYMENT_STATUS_UPDATES (PAYMENT_ID, NEW_STATUS, REASON)
SELECT PAYMENT_ID, 'COMPLETED', 'Cleared by beneficiary bank'
FROM RAW.PAYMENTS WHERE STATUS = 'PENDING' LIMIT 5;

-- Step 4: Stream captured it
SELECT PAYMENT_ID, NEW_STATUS, METADATA$ACTION FROM RAW.STATUS_STREAM;

-- Step 5: Fire the Task
EXECUTE TASK RAW.APPLY_UPDATES;

-- Step 6: Prove it worked
SELECT STATUS, COUNT(*) FROM RAW.PAYMENTS GROUP BY STATUS;
Event-driven pipeline in 4 SQL statements. No Airflow, no Lambda, no polling — zero cost when no data arrives.

4e. Stored Procedures — control flow

sql
CREATE OR REPLACE PROCEDURE PREPARED.FLAG_AGED_FAILURES()
    RETURNS VARCHAR LANGUAGE SQL EXECUTE AS CALLER
AS BEGIN
    LET cnt INTEGER := 0;
    SELECT COUNT(*) INTO :cnt FROM RAW.PAYMENTS
    WHERE STATUS = 'FAILED' AND PAYMENT_DATE < DATEADD(HOUR, -24, CURRENT_TIMESTAMP());
    IF (:cnt > 0) THEN RETURN :cnt || ' aged failures — flag for review';
    ELSE RETURN 'All clear'; END IF;
END;
CALL PREPARED.FLAG_AGED_FAILURES();
Stored Procedures handle branching, loops, and error handling. Useful for operational processes and conditional pipeline routing.

Cortex AI Enrichment

20 min — live SQL

What this covers: AI functions built directly into SQL — sentiment analysis, text classification, and LLM-generated recommendations. Plus AI-powered Dynamic Tables that refresh automatically.

Why it matters: No separate ML platform, no data movement, no model training. One function call on any text column. Same RBAC, same audit trail.

🔒
Security:

All Cortex AI runs inside Snowflake. Data never leaves the platform.

5a. Sentiment Analysis

sql
SELECT
    LEFT(FEEDBACK_TEXT, 80) || '...' AS PREVIEW,
    ROUND(SNOWFLAKE.CORTEX.SENTIMENT(FEEDBACK_TEXT), 3) AS SCORE,
    CASE WHEN SNOWFLAKE.CORTEX.SENTIMENT(FEEDBACK_TEXT) > 0.3 THEN 'POSITIVE'
         WHEN SNOWFLAKE.CORTEX.SENTIMENT(FEEDBACK_TEXT) < -0.3 THEN 'NEGATIVE'
         ELSE 'NEUTRAL' END AS LABEL
FROM RAW.CUSTOMER_FEEDBACK
ORDER BY SCORE LIMIT 10;
One function call scores any text from -1 (negative) to +1 (positive). No model training, no external APIs.

5b. Text Classification

sql
SELECT
    FEEDBACK_TEXT,
    SNOWFLAKE.CORTEX.CLASSIFY_TEXT(FEEDBACK_TEXT,
        ['Performance Issue','Service Quality','Technical Problem',
         'Positive Experience','Feature Request']
    ):label::STRING AS TOPIC
FROM RAW.CUSTOMER_FEEDBACK LIMIT 5;
Classify any text into your own categories — no training data needed. Define the labels, Cortex does the rest.

5c. LLM-Generated Actions

sql
SELECT ALERT_ID, SEVERITY, ALERT_DESCRIPTION,
    SNOWFLAKE.CORTEX.COMPLETE('mistral-large2',
        'You are a payments ops analyst. Given alert: "' ||
        ALERT_DESCRIPTION || '" severity: ' || SEVERITY ||
        '. One-sentence recommended action.'
    ) AS AI_ACTION
FROM RAW.OPERATIONAL_ALERTS
WHERE STATUS = 'OPEN' AND SEVERITY IN ('HIGH','CRITICAL')
LIMIT 5;
Any LLM, directly in SQL. Context-aware recommendations generated per row. Swap models with one word.

Streamlit App

5 min — live demo

What this covers: Streamlit in Snowflake — interactive data applications in Python, running natively inside Snowflake with no separate infrastructure.

Why it matters: Not everyone wants a SQL worksheet or BI tool. Streamlit lets you build custom dashboards and operational tools that sit on top of the data you've just been querying — governed, no servers, no data extraction.

Payments Health Monitor

A dark-themed operational dashboard querying the Dynamic Tables we built in Sections 4 and 5. Filters, KPI cards, failure rate charts, volume bars, and AI sentiment breakdown — all in ~100 lines of Python.

python
import streamlit as st
from snowflake.snowpark.context import get_active_session

session = get_active_session()
st.title("Barclays Payments Health Monitor")

df = session.sql("""
    SELECT PAYMENT_DAY, PAYMENT_TYPE, TXN_COUNT, TOTAL_VALUE, FAIL_RATE_PCT
    FROM PREPARED.DAILY_SUMMARY
    ORDER BY PAYMENT_DAY DESC LIMIT 60
""").to_pandas()

col1, col2, col3 = st.columns(3)
col1.metric("Transactions", f"{df['TXN_COUNT'].sum():,}")
col2.metric("Value", f"£{df['TOTAL_VALUE'].sum():,.0f}")
col3.metric("Fail Rate", f"{df['FAIL_RATE_PCT'].mean():.1f}%")

st.line_chart(df, x="PAYMENT_DAY", y="FAIL_RATE_PCT")
Full circle: RAW → View → Dynamic Table → Streamlit app. All governed, all inside Snowflake, shareable like any other object.
